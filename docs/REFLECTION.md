## 二、性能实测（10 万行）

```
扫描行=100002 产出行=100000
流式解析=3527ms  解析+校验+聚合=3621ms
有效=100000 错误=0 出库单=12500
通过服务器端到端：preview ~4.3s，import(12500单) ~4.2s
```

内存：流式引擎本身恒定内存（逐行回调，不驻留整本工作簿）；测试脚本因收集全部行
作检查才升到 ~200MB 堆。生产导入路径边读边聚合入库，不需要一次性持有全部行。

## 三、AI（Claude Code）协作反思

### 做得好的地方
- **架构决策**：很快确立「LLM 只生成规则、确定性引擎跑全量」的分离架构，这同时
  解决了正确性（确定性、可复用）和性能（不让 LLM 碰 10 万行）两个考点，是本方案的命脉。
- **真实数据驱动**：先把 7 个考试附件全部解析、看清每种非标形态的真实行列结构，
  再设计 DSL 和引擎，而不是凭空设计。每种格式都写了手写规则做回归测试，边写边验证。
- **快速自愈**：引擎首跑时配送发货单的表头行号偏移、keyValue 子串误命中（「收货人」
  命中「收货人手机号」）、PDF 折行丢数量等问题，都通过打印真实网格定位、针对性修复。

### 做得不够/踩的坑
- **图片读取**：考试要求是 PNG 截图，当前环境的图片视觉读取始终返回空，绕了不少弯
  （转 JPEG、缩放都无效），最终用 Windows OCR（winocr）才提取出文字。OCR 对模糊的
  「功能需求」「评分标准」两页识别质量差，部分细则只能靠上下文推断。**这是最大的时间损耗点。**
- **DSL 表达边界**：cardRepeat 块的明细列映射目前复用全局 fieldMappings 的 column
  来源（卡内明细表头列固定时可行），若卡片内表头列顺序多变则需扩展；transpose 块假设
  门店列连续，遇到门店列不连续需要改进。
- **PDF 解析脆弱性**：依赖 pdf-parse 的纯文本抽取 + 正则锚点，对「编码正则 + 行尾数字」
  模式有效，但若 PDF 是扫描件或多列复杂排版会失效，生产需引入更强的 PDF 表格还原。

### 节省了多少时间
若纯手写，这套含提取层 / DSL / 引擎 / 双驱动 DB / 全套 UI / 文档的全栈项目，
估计需 2–3 个工作日。借助 AI 在数小时内完成骨架到验证全流程，**大致节省 70%+ 时间**，
省下的主要是样板代码、类型定义、跨格式边界处理的试错。

### 如果让人工接手优化
- 给 PDF/Word 引入更可靠的表格结构还原（如基于坐标的单元格聚类）。
- cardRepeat/transpose 支持卡内独立列映射与非连续门店列。
- 大文件导入改为「上传到对象存储 + 后台队列流式入库 + 进度回传」，绕开 Serverless body 限制。
- 规则版本管理与 A/B 试运行对比。

## 四、与 V1 的核心变化
1. UI 与下单系统保持一致的设计语言（主色卡片风格）。
2. 通过大模型实现解析规则的自动生成（V1 需人工为每种文件编码解析逻辑）。
3. 新增严苛的性能要求（10 万行级流式处理）。
